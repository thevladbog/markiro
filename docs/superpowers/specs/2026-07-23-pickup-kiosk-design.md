# Самообслуживаемый киоск «Для себя» — дизайн-спека

> Статус: утверждён к планированию (2026-07-23).
> Источники: `docs/design-briefs/design_handoff_markiro/` (прототипы
> `prototypes/pickup-kiosk.dc.html`, `prototypes/pickup-slip.dc.html` и раздел
> README про «Для себя»), `docs/architecture.md`, действующий код
> (`packages/domain`, `packages/db`, `apps/api`, `apps/admin`).
> Следующий шаг: writing-plans → нумерованный план в `docs/superpowers/plans/`.

## 1. Обзор и цель

Отдельный сенсорный терминал самообслуживания у проходной цеха/склада.
Сотрудник сканирует **QR-код своего бейджа**, затем сканирует **коды маркировки
«Честный ЗНАК» (DataMatrix / КМ)** с единиц продукции, которые забирает для
себя. Формируется **заявка** (`ORD-ГГ-НННН`), которую администратор в админ-панели
проводит одним из способов: **«Пробита на кассе»** (продажа сотруднику, чек ККТ)
или **«Списана актом»** (списание на расходы, подпричина). Markiro в v1 —
слой **фиксации + документа + статуса**; реальный вывод кодов из оборота
происходит во внешних системах клиента (касса ККТ→ОФД→ГИС МТ; акт в ГИС МТ/1С).

## 2. Зафиксированные решения (decision log)

1. **Платформа киоска** — отдельное веб-приложение `apps/kiosk` (не режим станции,
   не маршрут в админке). Цель: планшеты Android/Apple или сенсорные моноблоки
   Windows/Linux.
2. **Ввод сканера** — плагаемый транспорт: **Web Serial** (где доступен — Chrome/Edge
   на десктоп-моноблоке, сырые байты с разделителем GS) + **клавиатурный
   (HID/BLE) фолбэк** (везде, включая планшеты). Поверх — обязательный
   **структурный, group-aware KM-guard** на базе `packages/domain`.
   `]d2` (AIM-префикс DataMatrix) **не обязателен** — детект структурный.
3. **Проведение заявки** — фиксация + ведомость + статус. **Без прямой
   интеграции ГИС МТ/СУЗ** (это отдельная поздняя фаза с УКЭП, см. architecture
   «Open items»). Интеграция с ККТ не делается — номер чека вводится вручную.
4. **CommerceML** — **отложен в отдельный план**, но в этом плане закладывается
   готовность (внешние ID товаров, поля цены, экспортируемая структура заявки).
5. **Offline** — киоск офлайн-способен: локальный кэш справочников + буфер
   заявок + фоновый синк (в духе станции).
6. **Сотрудники/бейджи** — канонический минимальный реестр строится **в этом
   плане**; станция (план 05) его расширяет, а не дублирует (см. §4).
7. **Несуществующий/недоступный товар** при скане — **красная модалка**
   «Товара нет в каталоге, обратитесь к администратору».
8. **Подпричины списания** — настраиваемый список на тенант (редактор в админке),
   дефолт: Бой / Маркетинг / Подарок / Другое.
9. **Нумерация заявок** — сквозной счётчик на тенант, формат `ORD-ГГ-НННН`
   (НННН с лидирующими нулями, ГГ = год создания).

## 3. Границы

**В v1:** приложение киоска (3 экрана, 2 ориентации, offline), реестр сотрудников

- QR-бейджи, киоски + настройки, заявки и их жизненный цикл, раздел «Для себя» в
  админке (свод с фильтрами, карточка заявки, массовая выгрузка кодов), печатная
  ведомость A4, поля цены/ЕГАИС в карточке товара, доменный KM-guard.

**Вне v1:** прямой вывод из оборота через ГИС МТ/СУЗ (УКЭП); CommerceML-обмен
(импорт цен, экспорт заявок); интеграция с фискальным регистратором (ККТ).

## 4. Место в роадмапе и контракт «сотрудники/бейджи»

Новый самостоятельный план. Зависит от плана 03 (каталог — готов) плюс добавка
полей в `products`. Независим от станции (05) и агрегации (06); может исполняться
параллельно станционному треку. В индекс `docs/superpowers/plans/2026-07-21-markiro-mvp-roadmap.md`
добавляется запись.

**Контракт переиспользования.** Ни `employees`, ни бейджей в коде сейчас нет.
Этот план создаёт канонический **минимальный** реестр (`employees`,
`employee_badges`) в `packages/db` + API + управление в админке. Станция (план 05,
офлайн PIN/бейдж-вход) **расширяет** эти таблицы (`pinHash`, поля офлайн-синка), не
создаёт свои. Если реестр сотрудников делается параллельно другим планом/агентом —
интерфейс единый (одни и те же таблицы/эндпоинты), кто первый реализует, тот и
владеет схемой; второй консюмер только читает. Это явная точка синхронизации.

## 5. Архитектура компонентов

- **`apps/kiosk`** (новое) — React 19 + Vite 8 + TS, тёмная тема из `packages/ui`
  (floor-компоненты). Онлайн-first с offline-буфером. Вход по device-token
  (enrollment, аналогично станции). Слой «источник сканирования» (клавиатура +
  Web Serial). PWA (service worker, шрифты bundled, без CDN).
- **`packages/domain`** — хардинг парсера КМ + `validatePickupKm()` (group-aware),
  `kmKey()` для дедупа (уже есть). См. §7.
- **`packages/ui`** — при необходимости добавить недостающие floor-компоненты,
  используемые киоском (по прототипу).
- **`apps/api`** (NestJS) — device-auth киоска; эндпоинты киоска и админки. См. §9.
- **`apps/admin`** — раздел «Для себя», настройки киосков, управление сотрудниками,
  поля в карточке товара, массовая выгрузка, генерация ведомости.
- **Рендер штрихкодов** — новая зависимость **bwip-js** (bundled, без CDN):
  DataMatrix (карточка заявки 64px, ведомость 13мм), QR (бейдж), Code128 (номер
  заказа в подвале ведомости). Разместить в `packages/domain` или `packages/ui`
  (уточнить при планировании — там, где нет утечки серверных/браузерных API).

## 6. Модель данных

Все таблицы tenant-scoped; композитные FK `(tenant_id, id)` как в
`packages/db/src/schema/platform.ts`.

- **`employees`** — `id, tenant_id, full_name, role, status(active|archived),
created_at`. (Станция позже добавит `pin_hash` и поля синка.)
- **`employee_badges`** — `id, tenant_id, employee_id, badge_code(uniq per tenant),
label, issued_at, revoked_at`. Отдельная таблица ради перевыпуска/отзыва
  (потеря бейджа). `badge_code` = payload QR-бейджа.
- **`kiosks`** — `id, tenant_id, name, location, device_token_hash,
day_limit_per_employee(int), show_prices(bool), last_seen_at, status, created_at`.
- **`kiosk_products`** — allowlist `(tenant_id, kiosk_id, product_id)` — «доступная
  продукция» киоска.
- **`pickup_order_reasons`** — настраиваемые подпричины списания на тенант:
  `id, tenant_id, name, sort_order, archived`. Сидируются дефолтом
  (Бой/Маркетинг/Подарок/Другое).
- **`pickup_orders`** — `id, tenant_id, order_no(text, ORD-ГГ-НННН, uniq per tenant),
kiosk_id, employee_id, reason(buy|writeoff), writeoff_reason_id(nullable),
status(pending|punched|writtenoff|cancelled), item_count, total_price
numeric(12,2) nullable, receipt_no(nullable), act_no(nullable), created_at,
resolved_at, resolved_by_user_id, device_seq(для идемпотентности синка)`.
- **`pickup_order_items`** — `id, tenant_id, order_id, product_id, gtin14, serial,
raw_km, km_key, unit_price numeric(12,2) nullable, scanned_at`.
  - `unique(tenant_id, order_id, km_key)` — нет дублей внутри заявки.
  - partial-unique `(tenant_id, km_key)` среди заявок в статусах ≠ cancelled — одна
    физединица не может быть в двух открытых заявках. Нарушение при синке →
    конфликт админу.
- **`products` +** `unit_price numeric(12,2) nullable`, `egais_code text nullable`,
  `external_ref text nullable` (1С GUID — CommerceML-готовность).

**Счётчик номеров** — per-tenant, присваивается сервером в транзакции при приёме
заявки (см. §13). Офлайн-заявки получают реальный `order_no` при синке.

## 7. Доменный слой: KM-guard

Опираемся на существующее:

- `parseKm()` (`packages/domain/src/gs1/km.ts`): `01<gtin14>21<serial><GS>…`,
  опциональный `]d2`, сбор хвостовых AI (91/92/93) по GS, проверка контрольной
  цифры GTIN.
- `classifyScan()` (`packages/domain/src/scan/classify.ts`): SSCC / GTIN / KM /
  unknown — структурный детект (различает бейдж и КМ).
- `kmKey()` = `01<gtin>21<serial>` — канонический дедуп.

Добавляем:

1. **Group-aware валидация** `validatePickupKm(raw, opts?)` — для известных
   товарных групп (алкоголь: хвост AI `93`, 4 символа, GS перед `93`) проверяет
   ожидаемую структуру хвоста.
2. **Закрыть гочу**: текущий `parseKm` при **отсутствии GS** молча забирает весь
   хвост в серийник → неверный `km_key`. Guard должен это детектировать
   (подозрительно длинный серийник / отсутствие ожидаемого хвоста / GS не там) и
   возвращать статус «прочитано не полностью» вместо тихого принятия. Киоск на
   такой статус показывает «Код прочитан не полностью, поднесите ещё раз».
3. Все изменения — обратносовместимы для станции (`scan/validate.ts`,
   `scan/classify.ts` — не ломать текущие контракты; расширять, а не менять).

## 8. Приложение киоска (`apps/kiosk`) + offline

**Экраны** (по `pickup-kiosk.dc.html`), 2 ориентации (альбом 1180×800 / вертикаль
800×1180):

1. **Ожидание** — «Возьмите продукцию для себя», зона скана QR-бейджа.
2. **Отбор** — шапка (лого, ФИО+аватар, «Не я»); зона скана КМ; список «Вы берёте»
   (фото, название, хвост кода, цена, удаление); лимит шт/день с остатком; причина
   Покупка (дефолт) / Списание (+обязательная подпричина-чип); «Готово — передать
   администратору». Повторный код → янтарный баннер. Лимит достигнут → блокирующая
   зона. Несуществующий/недоступный товар → **красная модалка**.
3. **Передано** — «Заявка № N передана», сводка (причина·штук·сумма), автосброс 10 с.

**Источник сканирования** — абстракция с двумя транспортами:

- **Клавиатурный listener** (буфер до терминатора Enter/таймаут) — универсален.
- **Web Serial** — где `navigator.serial` доступен; сырые байты с GS; авто-reconnect.
- Каждое событие → `classifyScan` + `validatePickupKm`; бейдж (QR) отличается от КМ
  структурно. Демо-кнопки прототипа заменяются реальными событиями сканера.

**Offline-модель**:

- **Локальный кэш (IndexedDB)**: сотрудники + бейджи, доступные товары (gtin, имя,
  цена, фото, ЕГАИС), конфиг киоска (лимит, показ цен, подпричины). Тянется
  **периодически при онлайне + при старте**.
- **Локальная валидация офлайн**: бейдж→сотрудник, KM-guard, дедуп внутри буфера,
  дневной лимит best-effort (по локально известным заявкам).
- **Буфер + синк**: заявки в очереди, отправка идемпотентная (device-sequenced,
  `device_seq`) при онлайне.
- **Авторитет — сервер**: реальный `order_no` присваивается при приёме;
  кросс-киосковый дедуп (partial-unique `km_key`) и настоящий дневной лимит
  проверяются на сервере; конфликты (код дважды / превышен лимит) всплывают админу.
  Офлайн-экран «Передано» показывает локальное подтверждение; № появляется после
  синка.
- **Статус в сети** киоска в админке — по `last_seen_at` (heartbeat/синк).

## 9. API (NestJS)

**Киоск-facing (device-auth по токену enrollment):**

- Получить конфиг киоска (лимит, показ цен, подпричины, allowlist).
- Периодическая выгрузка справочника сотрудников+бейджей и доступных товаров+цен
  (для офлайн-кэша); поддержка инкремента/ETag по возможности.
- Разрешить бейдж → сотрудник.
- Валидация/резолв скана КМ → товар (allowlist) + цена + проверка дубля/лимита
  (онлайн). Ответ на неизвестный/недоступный товар — машиночитаемый код ошибки
  под красную модалку.
- Приём заявки (идемпотентно по `(kiosk_id, device_seq)`), присвоение `order_no`,
  ответ с конфликтами (дубли/лимит).
- Heartbeat / `last_seen_at`.

**Админ-facing (сессия Better Auth):**

- Заявки: список с фильтрами (статус, причина, дата), карточка, «провести»
  (punched: `receipt_no` / writtenoff: `act_no` + подпричина), «отменить».
- Массовая выгрузка кодов: по выбранным заявкам → `.txt` UTF-8 (см. §12).
- Генерация печатной ведомости по заявке (§11).
- Сотрудники: CRUD; выпуск/отзыв QR-бейджа.
- Киоски: CRUD; enrollment-токен; allowlist товаров; лимит; показ цен; статус.
- Подпричины списания: CRUD (`pickup_order_reasons`).
- Поля товара: `unit_price`, `egais_code`, `external_ref`.

Везде — tenant-изоляция (как в существующих модулях).

## 10. Админка «Для себя»

- **Сайдбар** — пункт «Для себя», бейдж = число заявок в статусе `pending`.
- **Свод заявок** — **фильтры по статусу** (Все/Ожидают/Пробиты/Списаны/Отменены)
  **+ по причине** (Покупка / Списание·подпричина) **+ дата**. Колонки: № (`ORD-…`),
  сотрудник, киоск, когда, продукция, причина, кодов, сумма, статус, «Печать».
  Режим **«Массовая выгрузка»** → мультивыбор заявок → «Выгрузить коды».
- **Карточка заявки** — QR бейджа + сотрудник/киоск/причина/создана/итог; список
  кодов: DataMatrix 64px (скан с экрана), полный КМ моноширинным, цена, статус
  «в обороте»; действия: «Пробита на кассе» (номер чека), «Списать актом» (номер
  акта), «Печать», «Отменить».
- **Управление сотрудниками** — список + карточка; выпуск/отзыв QR-бейджа.
- **Настройки → Киоски** — список киосков (статус в сети по `last_seen_at`),
  enrollment-токен, allowlist товаров, лимит/сотрудник, показ цен; **редактор
  подпричин списания**.
- **Карточка товара (каталог)** — поля «Цена за шт., ₽», «Код ЕГАИС»; `external_ref`
  скрытое/служебное.

Все UI: RU primary + EN, светлая/тёмная темы, дизайн-система Markiro.

## 11. Печатная ведомость отбора (A4)

Серверный print-ready HTML/PDF по конкретной заявке (по `pickup-slip.dc.html`):
шапка (лого + «Ведомость отбора по заявке № N»); блоки организация / сотрудник /
заявка (с причиной); таблица КМ (№, продукт, полный КМ, цена, **DataMatrix 13мм**,
пригоден для скана на кассе); итог; блок «Способ вывода из оборота» (чекбоксы: чек
ККТ № / акт списания № + подпричина; строка с причиной, выбранной сотрудником);
блок с **QR бейджа 22мм** «Отсканируйте, чтобы найти сотрудника»; подписи
«Продукцию получил» / «Администратор»; футер.
**Дополнительно к прототипу:** в подвал — **Code128 с номером заказа `ORD-ГГ-НННН`**
(штрихкодируем, по аналогии с остальными документами). Все штрихкоды/QR — из
реальных данных через bwip-js.

## 12. Массовая выгрузка кодов

Из свода: режим «Массовая выгрузка» → выбор заявок → «Выгрузить коды» →
файл **`.txt` в UTF-8, по одному КМ (payload DataMatrix) на строку**, сырой код как
сохранён (`raw_km`). Разделитель GS (0x1D) внутри КМ **сохраняется как есть**.
_Открытый вопрос: точное представление GS в txt под приёмник клиента (ГИС МТ / 1С) —
подтвердить при планировании; дефолт — литеральный 0x1D._ Имя файла — напр.
`codes-YYYYMMDD.txt`.

## 13. Нумерация заявок

Формат `ORD-ГГ-НННН`, напр. `ORD-26-0037`. Счётчик **сквозной на тенант**, НННН с
лидирующими нулями (4 знака; при переполнении 9999 — расширяется). ГГ = год
создания. Присваивается сервером в транзакции при приёме заявки (в т.ч. при синке
офлайн-заявок), чтобы номера были плотными и без коллизий.
_Допущение: без ежегодного сброса НННН (сквозной). Если нужен сброс по годам —
счётчик per (tenant, year); фиксируется при планировании._

## 14. CommerceML-готовность (только хуки)

`products.external_ref` (1С GUID), поля цены на товаре и позиции заявки,
устойчивая экспортируемая структура заявки. Реализация обмена (импорт цен,
экспорт заявок как документов 1С) — отдельный будущий план, подключается адаптером
(architecture §5 «format adapters»).

## 15. Тестирование (TDD)

- **domain**: KM-guard — пропавший GS, алкогольный хвост `93×4`, `km_key`,
  нормализация; обратная совместимость станционных контрактов.
- **api**: жизненный цикл заявки; дедуп внутри заявки и кросс-киосковый; дневной
  лимит; идемпотентный синк + конфликты; device/бейдж-auth; tenant-изоляция;
  генерация `ORD-…`; массовая выгрузка (формат txt/UTF-8).
- **kiosk**: источник сканирования (клавиатура/Web Serial), офлайн-очередь и синк,
  экраны и переходы (vitest + RTL), интеграция `classifyScan`.
- **admin**: свод + фильтры (статус/причина/дата), массовая выгрузка, карточка +
  действия, управление сотрудниками/киосками, поля товара.

## 16. Открытые вопросы / допущения (перенести в план)

1. Представление GS в txt-выгрузке (§12) — литеральный 0x1D по умолчанию.
2. Сброс счётчика заявок по годам (§13) — по умолчанию нет.
